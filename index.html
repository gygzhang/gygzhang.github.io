实验简介

首先是项目地址： xv6 mmap lab

本实验要求我们在xv6中实现mmap系统调用，mmap主要功能是将文件和内存建立一种映射，当我们修改想要对文件进行读写时，就不需要使用write/read等系统调用，而是通过读写内存来实现读写文件。

那么为什么要使用mmap，而不是直接读取文件呢？原因在于，以read系统调用为例，我们使用mmap，能够减少从内核空间向用户空间复制数据的次数。我们可以看一下，read系统调用中调用到的一个函数：

从上面的函数可以看到，如果从用户空间调用read，那么操作系统将调用copyout将数据从内核空间复制到用户空间（read从磁盘中读取数据到内核空间，且用户不能直接访问内核空间）。

而我们使用mmap，对文件的读写，都是直接对用户空间的内存进行操作的，这样自然就避免了一些不必要的开销。

后记

本次实验，其实不难，只要理解了mmap的思想和流程，实验就能很容易的做出来。不过由于距离上个实验过去了很久，导致对xv6的一些细节已经不是很熟悉了。比如，mmap传入的是一个文件描述符，那么怎么获得对应的file结构体呢？ 答案是每个进程都有一个打开文件集合，从而可以利用fd在该集合中寻找对应的file结构体。当然也有一些东西，通过这次实验也加深了印象，比如关于页表机制，xv6采用三级页表，也就是说，给定一个虚拟地址va，需要经过三次运算才能得到真正的物理地址pa。具体过程为，首先我们需要知道riscv实际上还是存在更多级的页表的（比如四级页表），因此64位的虚拟地址有一些位是用不到的，对于xv6采用的三级页表模式，有3*9=27位的索引位和12位的页内偏移位，一共39位，这在riscv中被称为sv39。27位的前9位索引位会首先在进程的页表中（页表项数为2^9=512）定位一个页表项PTE3，该页表项是一个地址，指向了下一级的页表，27位索引位中的中间9位用来定位这个页表的某一个页表项，该项仍然是一个地址，指向第三级的页表。这些页表项的只有前54位被用到，其中后10位是PTE的标志位，指示该页能否被读写执行等。剩下的44位被用于寻找下一级页表（前两级页表），或者定位物理内存（最后一级页表）。当定位物理时， pa= PPN + offset = (( *PTE3>>10 ) << 12) + va&0x3FF 。 当然本次实验也还有更多高难度的附加题，时间关系，只能等以后有需要的时候再来挑战了。

任务一： Memory allocator

描述

在xv6中，物理内存都是由操作系统实现的内存分配器管理的，内核通过申请物理内存，然后将其映射到虚拟内存，提供给用户进程使用。而要申请物理内存，就要使用内存分配器提供的函数kalloc，要释放物理内存，使用kfree来进行释放。

任务一主要要解决的问题是多核CPU之间同时使用内存分配器的冲突。
在OS启动的时候，会调用一个物理内存的初始化函数kinit()，该函数将可给用户进程使用的物理内存以4096字节为一页，组织成页的集合，然后将这些页连接成一个链表，这个链表就是kalloc和kfree操作的对象。
当多个CPU都需要给获取物理内存的时候，访问的是同一个链表，因此需要通过锁来进行共享资源的保护，这样会将不同CPU的分配申请串行化。

当进程想要访问磁盘数据，就需要使用以上两个数据结构，通过检查buf中的valid域，来判断是否需要再次从磁盘中读取数据，通过refcnt域来判断该buf是否没有被任何进程所引用。bget的两个参数通过与buf中对应参数进行比较，如果相等，则不需要再次从磁盘中读取数据，否则应该从磁盘中读取数据(通过valid域实现)。

xv6是使用LRU最近最少使用算法来实现的，但是buffer cache只有一个，在不同cpu、不同进程之间都存在锁的竞争，给每个CPU单独的buffer cache，不同cpu上的进程可能会访问同一个buffer cache，这个会使实现变得非常复杂，而且性能可能会更差，因此不能像kalloc那样通过实现per-CPU内存分配器就能消除大部分的锁竞争。因此lab建议我们使用哈希桶来解决锁竞争。

实现思路

因此解决方案是定义一个哈希表，内部每个哈希桶都有自己的锁，有自己的buf，磁盘号通过哈希函数

来映射到不同或相同的哈希桶中，为什么这样能减少锁竞争呢？ 想象一下，如果没有哈希桶，两个本来应该映射到不同哈希桶的磁盘号，就会通过同一个buffer cache来进行访问。如果他们同时调用bget函数，这两个磁盘号之间就会存在锁竞争，而如果使用哈希表，这两个磁盘号就会映射到不同的哈希桶之中，他们可以独立进行操作。

刚刚提到如果指定socket对应的接收队列为空，那么它怎么被唤醒的呢？

为了搞清楚这个问题，我们首先来看网卡驱动程序，网络属于外部设备，所以如果有数据到来，网卡设备会以中断的方式通知CPU，cpu则会调用中断处理函数，网卡的中断处理函数为e1000_intr(),定义如下：

可以看到，中断函数将调用e1000_recv来接收到来的数据，这也是xv6实验要求我们实现的函数之一，通过查询手册，该函数实现如下：

该函数用到了两个关键的数据结构。一个是rx_ring(接收环)的数组，其元素是rx_desc(接收描述符)，这两个数据结构都是和e1000网卡密切相关的，rx_ring就是一个循环数组，rx_desc内部有很多field，e1000网卡接收到数据之后，数据便是储存在rx_desc之中的。

要读取数据，需要用到E1000_RDT这个寄存器，手册里面说的：
HARDWARE OWNS ALL DESCRIPTORS BETWEEN [HEAD AND TAIL].
结合HEAD和TAIL寄存器的说明，我们应该将HEAD初始化为0，TAIL初始化为环形数组的最后一个位置，这样硬件能将到来的网络包储存在HEAD和TAIL之间，如果收到一个packet，硬件就会将它放到HEAD指向的位置，由此应该用(regs[E1000_RDT]+1)%RX_RING_SIZE来读取接收到的数据包。

重点是，网卡一次可能接收到了很多包，所以e1000_recv函数应该遍历在TAIL->HEAD之间所有的描述符，直到遇到一个描述符的状态域的E1000_RXD_STAT_DD为0，则表示所以数据都被读取完毕，应该返回。

总结

通过上面分析可以发现，如果getc函数会使_IO_file_ptr加一，类似于：

而ungetc()则分两种情况：

  1. ungetc字符和getc()读取的字符(或者_IO_file_ptr指向字符的前一个字符)一样,那么该函数将_IO_file_ptr减一
  2. 如果ungetc传入的字符与getc()读取的字符(或者_IO_file_ptr指向字符的前一个字符)不一致，那么_IO_file_*系列指针都将指向另一片不同的内存区域，_IO_file_ptr指向的字符正是ungetc传入的字符。

需要注意，ungetc()改变了_IO_file_* 系列指针，GLIBC 肯定是做了标记的，再读取完之后，IO_file* 指针将变回原来的值，以保持对程序员的透明。
